# Життєвий Цикл Турніру та Раундів (State Machine)

Цей документ описує актуальну логіку переходів статусів для `Tournament` та `Round`.

---

## 1. Статуси Турніру (Tournament)

Можливі статуси: `draft`, `registration`, `running`, `finished`.

### Переходи статусів
1. **`draft` -> `registration`**
- **Хто ініціює:** адміністратор.
- **Як:** `POST /api/tournaments/{id}/start-registration/` (`start_registration`).
- **Guard:** лише турнір у статусі `draft`.

2. **`registration` -> `running`**
- **Хто ініціює:** адміністратор, запускаючи перший раунд.
- **Як:** `POST /api/tournaments/rounds/{id}/start/` (`start_round`).
- **Логіка:** при успішному старті раунду турнір переводиться у `running`.

3. **`running` -> `finished`**
- **Хто ініціює:** автоматично в сервісній логіці.
- **Як:** після `mark_round_evaluated()` або під час `sync_time_based_statuses()`.
- **Guard:** у турнірі не залишилось раундів зі статусом, відмінним від `evaluated`.

---

## 2. Статуси Раунду (Round)

Можливі статуси: `draft`, `active`, `submission_closed`, `evaluated`.

### Переходи статусів
1. **`draft` -> `active`**
- **Хто ініціює:** адміністратор.
- **Як:** `POST /api/tournaments/rounds/{id}/start/`.
- **Guards (`start_round`):**
- раунд має бути у статусі `draft`;
- має настати час старту: `start_date <= now`;
- дедлайн не має бути в минулому: `end_date > now`;
- турнір має бути у статусі `registration` або `running`;
- якщо раунд є першим за хронологією (`найменший start_date`), турнір повинен бути у `registration`;
- якщо раунд не перший, найближчий попередній раунд (за `start_date`) має бути `evaluated`;
- в турнірі не повинно бути іншого `active` раунду.

2. **`active` -> `submission_closed`**
- **Хто ініціює:** автоматично за часом або адміністратор вручну.
- **Автоматично:** `sync_time_based_statuses()` закриває активні раунди з `end_date <= now`.
- **Вручну:** `POST /api/tournaments/rounds/{id}/close-submissions/`.
- **Guard для ручного закриття:** лише `active` раунд.

### Призначення журі в `submission_closed`
- Після переходу в `submission_closed` призначення журі виконується **вручну адміністратором** через:
- `POST /api/evaluation/rounds/{id}/assign-jury/`.
- Автоматичний/рандомний розподіл не використовується.
- Поточна модель: frontend передає повний список призначень; backend валідовує і повністю замінює призначення раунду.

3. **`submission_closed` -> `evaluated`**
- **Хто ініціює:** адміністратор вручну або автоматично при завершенні оцінювання журі.
- **Вручну:** `POST /api/tournaments/rounds/{id}/mark-evaluated/`.
- **Guard для ручного переходу:** раунд має бути у `submission_closed`.
- **Автоматично:** після оцінювання, коли для всіх призначень журі є оцінки.

---

## 3. Бізнес-правила Раундів (create/update)

Порядок раундів визначається тільки `start_date` (поля `position` немає).

Валідація виконується через `Round.clean()` + serializer-guards:
- `start_date < end_date`;
- дати раунду в межах дат турніру;
- раунди одного турніру не можуть перетинатися в часі;
- приклад забороненого перетину: `round1(21.04-24.04)` та `round2(23.04-27.04)`;
- `winners_count` дозволено лише для останнього раунду;
- якщо у турнірі лише 1 раунд, його дати повинні збігатися з датами турніру;
- `passing_count` не може перевищувати кількість зареєстрованих команд (якщо реєстрації вже є).

Додатково:
- `DELETE /api/tournaments/rounds/{id}/` заборонено для останнього раунду турніру.

---

## 4. Що робить `sync_time_based_statuses()`

- Переводить `active` раунди з простроченим дедлайном у `submission_closed`.
- Перевіряє `running` турніри і переводить їх у `finished`, коли всі раунди вже `evaluated`.
- Не запускає раунди автоматично (`draft` -> `active` не робить).

---

## 5. Критичні Guard-правила для Submission

Подача/оновлення `Submission` дозволені лише коли:
- раунд має статус `active`;
- поточний час `now < round.end_date`;
- команда має активну реєстрацію в турнірі (`TournamentTeamRegistration.is_active = true`);
- користувач є капітаном або учасником цієї команди.
