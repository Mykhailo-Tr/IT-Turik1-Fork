# Життєвий Цикл Турніру та Раундів (State Machine)

Цей документ описує логіку переходів статусів (state machine) для Турнірів та Раундів.

---

## 1. Статуси Турніру (Tournament)

Можливі статуси: `draft`, `registration`, `running`, `finished`.

### Переходи статусів:
1. **`draft` $\rightarrow$ `registration`**
   - **Хто ініціює:** Тільки Адміністратор.
   - **Як:** Виклик ендпоінту `POST /api/tournaments/{id}/start-registration/`.
   - **Логіка:** Турнір переходить у фазу реєстрації, що дозволяє командам подавати заявки.

2. **`registration` $\rightarrow$ `running`**
   - **Хто ініціює:** Адміністратор (непрямо, запускаючи перший раунд).
   - **Як:** При виклику `start_round()` для першого раунду (`position == 1`).
   - **Логіка:** Автоматично встановлюється під час старту 1-го раунду.

3. **`running` $\rightarrow$ `finished`**
   - **Хто ініціює:** Автоматично (системою або при оцінці фінального раунду).
   - **Як:** При виклику `mark_round_evaluated()` або `sync_time_based_statuses()`.
   - **Логіка:** Якщо всі раунди турніру отримали статус `evaluated`, турнір вважається завершеним і автоматично переходить у `finished`.

---

## 2. Статуси Раунду (Round)

Можливі статуси: `draft`, `active`, `submission_closed`, `evaluated`.

### Переходи статусів:
1. **`draft` $\rightarrow$ `active` (Старт раунду)**
   - **Хто ініціює:** Тільки Адміністратор.
   - **Як:** Виклик `POST /api/tournaments/rounds/{id}/start/` (виклик функції `start_round`).
   - **Жорсткі перевірки (Guards):**
     - Раунд повинен мати статус `draft`.
     - Час: `start_date <= зараз < end_date` (час початку настав, але дедлайн ще не пройшов).
     - Турнір має бути в статусі `registration` або `running`.
     - Якщо це перший раунд (`position == 1`): турнір має бути в `registration`.
     - Якщо це наступний раунд (`position > 1`): попередній раунд (`position - 1`) **обов'язково** має бути `evaluated`.
     - В турнірі немає іншого `active` раунду (паралельні раунди заборонені).

2. **`active` $\rightarrow$ `submission_closed` (Закриття прийому робіт)**
   - **Хто ініціює:** Автоматично (або Адміністратор вручну).
   - **Автоматично:** Виклик `sync_time_based_statuses()` перевіряє всі активні раунди. Якщо `зараз >= end_date`, статус змінюється.
   - **Вручну:** Виклик `POST /api/tournaments/rounds/{id}/close-submissions/`.
   - **Логіка:** Після цього статусу команди більше не можуть надсилати або оновлювати свої роботи. Починається фаза роботи журі.

3. **`submission_closed` $\rightarrow$ `evaluated` (Фіналізація оцінок)**
   - **Хто ініціює:** Адміністратор (вручну) або **автоматично** (коли всі журі завершили оцінювання).
   - **Вручну:** Виклик `POST /api/tournaments/rounds/{id}/mark-evaluated/`.
   - **Автоматично:** Після створення/оновлення оцінки (`SubmissionEvaluation`) система перевіряє, чи всі `JuryAssignment` для раунду вже мають оцінки. Якщо так — раунд автоматично переводиться в `evaluated`.
   - **Важливий нюанс:** Авто-перехід виконується тільки якщо для раунду існує хоча б одне `JuryAssignment`. Якщо призначень немає (`0`) — автоматична фіналізація не спрацьовує.
   - **Логіка:** Адміністратор фіналізує результати раунду. Якщо це був останній раунд, цей крок також переведе весь турнір у статус `finished`.

---

## 3. Схема Роботи `sync_time_based_statuses()`

Ця функція відповідає за **автоматичні** переходи на основі часу. Її можна викликати за допомогою Cron або як middleware при запитах.

**Що вона робить:**
- Знаходить всі раунди `active`, для яких пройшов `end_date`, і переводить їх у `submission_closed`.
- Знаходить всі турніри `running`, перевіряє всі їхні раунди, і якщо абсолютно всі мають статус `evaluated`, переводить турнір у `finished`.
- **НЕ РОБИТЬ:** Автоматичний старт раундів (`draft` $\rightarrow$ `active`). Старт вимагає явного підтвердження адміністратора для контролю процесу.

---

## 4. Додаткові Guard-правила (не змінюють стани, але критичні для процесу)

- **Подача/оновлення роботи командою (`Submission`) дозволена тільки коли:**
  - раунд має статус `active`;
  - поточний час ще до дедлайну (`now < end_date`);
  - команда має **активну** реєстрацію в турнірі (`TournamentTeamRegistration.is_active = true`).
- **Призначення журі (`assign_submissions_to_jury`)** дозволене лише для раунду у статусі `submission_closed`.
- **Під час участі команди в активному турнірі** (турнір у `registration` або `running`) заборонені мутації складу/профілю команди:
  - створення інвайту до команди;
  - створення join-request;
  - прийняття інвайту;
  - прийняття join-request;
  - зміна `name`;
  - зміна `is_public`.
- **Виняток:** капітан може видалити учасника лише якщо поточна кількість учасників команди **строго більша** за `tournament.min_team_members` активного турніру.
